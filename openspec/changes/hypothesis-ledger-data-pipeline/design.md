## Context

The broader project turns coding-agent failure trajectories into training supervision. The motivation is empirical: the repo's earlier **oracle-free Elenchus experiments** showed that without an oracle the agent proposes weak hypotheses, self-confirms a hypothesis on the test result alone after editing (ignoring that the prior state may already have been correct), and lets band-aid / proxy patches pass as solved. An oracle is the only thing that can supply the corrective *direction* the agent cannot generate for itself — so the v1 pipeline uses one at collection time only, as a label midwife.

v1 is deliberately **simple**: a thin mini-SWE-agent (not the Elenchus engine) with one prompt change — state a hypothesis, then act, explore on your own. Infra reuse is limited to `run/swebench_single.py` (single-instance docker) and `scripts/_wincompat/` (Windows resource stub + CRLF fix). The earlier Elenchus engine machinery (matcher / residual / verifier-diagnostics / challenger, the C-differential) is out of scope — it was a workaround for not having an explicit direction; gold gives us one directly.

Four layers: Layer 0 raw trace → Layer 1 runtime ledger (the agent's own live hypotheses) → Layer 2 posthoc audit → Layer 3 training/eval views. Ledger ≠ Audit. Timeline is a goal, not a deadline.

## Goals / Non-Goals

**Goals:**
- A thin collection driver: mini-SWE-agent + hypothesis-first prompt + 3-stage no-early-oracle loop (explore → test feedback → oracle direction).
- A four-layer dataset with the runtime ledger sourced from the agent's live statements (no posthoc authorship).
- A posthoc audit + prefix-level training views (oracle-free input, minimal fixed-section target).
- A held-out turning-point eval anchored on gold/protocol turns, scored by a grader that uses a different model than the labeler.

**Non-Goals (v1) — reviewer gaps noted for later hardening:**
- The Elenchus engine (matcher / residual / promotion / verifier-diagnostics / challenger) and the C-differential.
- Grader human-κ calibration, positive/negative controls, bootstrap CIs.
- Base tool-use baselining, near-duplicate / diversity controls, per-sample-type floors, formal label-quality audits.
- Any model training, LoRA, DPO/RL; any SWE-bench resolve claim.

## Decisions

### D1 — Four layers; ledger ≠ audit; ledger = the agent's live hypotheses
Layer 1 cards capture only what the agent itself stated live during the run (its hypothesis, actions, observed evidence, self-verdict). No posthoc model writes a runtime field, and an allowlist rejects any field outside the runtime key set (so audit verdicts or renamed fields can't slip in). *Why:* keeps "what the agent knew" free of hindsight — the contamination that would otherwise flow into the oracle-free training input. *Alternative rejected:* one card holding both self-verdict and audit-verdict — that leak.

### D2 — Thin 3-stage no-early-oracle loop
A vanilla mini-SWE-agent with one prompt change: state a hypothesis, then act, explore on your own. On submit failure it gets the test output for a self-rescue; only if it still fails does an oracle (reading gold) say where it is wrong and the direction. Oracle before the agent's own submit is prohibited. *Why:* the agent must own its error path or there is no genuine turning point; the failure modes (weak hypo, self-confirm, band-aid) only surface when it explores unaided. *Alternative rejected:* oracle-on-first-wrong-hypothesis — no self-correction signal, and no engine machinery is needed for this.

### D3 — Oracle and all targets are direction-level (where-wrong + direction, never the answer)
The oracle redirect, and every training target, names where the reasoning is wrong and a probe/direction — never a patch, `file:line`, gold identifier, or hidden-test assertion. A simple leakage check (regex for `file:line`/diff + identifier overlap against the loaded gold) rejects offenders, backed by a sampled human spot-review (regex misses paraphrase). The runtime redirect is stored as a typed Layer 0 artifact and **fails loud** if it can't be produced. *Why:* a model trained on the answer learns "run toward gold", useless without gold at inference.

### D4 — Full-audit for labels, prefix for training; pre-submit target regenerated prefix-only
The strong model may read the full trajectory to find the turning point and write the Layer 2 audit; the exported sample contains only the prefix up to it. A pre-submit sample's target is regenerated by a separate strong-model call given only the prefix (blind to outcome), so it cannot launder a future test result into "weak support". *Why:* informed labels, honest runtime input. The runtime ledger being live-captured protects the input; this protects the target.

### D5 — Minimal fixed-section target; metadata out-of-band
Target = a `STATUS / WHY / RISK / MISSING EVIDENCE / NEXT MOVE / AVOID` text block — no nested JSON, candidate lists, or metadata. Provenance (`cut_step`, `leakage_level`) lives in a sidecar, never in the prompt/target; the input carries runtime facts only. *Why:* teaches the redirect policy, not a schema; fixed headers stay parseable for the grader. `AVOID` is phrased positively (the discriminating action to take instead).

### D6 — Gold/protocol-anchored, independent grader
Eval gold turning points come from protocol/harness events (the test-feedback turn, the oracle-redirect step, the visible-pass→PASS_TO_PASS-fail step), frozen, and the audit generator may not write them. The grader uses a model of a different version/family than the audit labeler, with rubric + scale frozen before scoring. (Heavier validity work — human-κ calibration, controls, CIs — is deferred to later hardening.) *Why:* otherwise the headline number is "the labeler model agreeing with itself".

## Risks / Trade-offs

- **Oracle redirect leaks the answer in prose** → direction-level check + identifier overlap vs gold + sampled human spot-review.
- **Oracle redirect silently no-fires** → fail-loud requirement + typed artifact + the in-loop sanity precheck.
- **In-loop harness eval corrupted on Windows (CRLF→all-0, codex-proxy effort floor)** → reuse `_wincompat` on PYTHONPATH; a precheck asserts a gold patch reports resolved=true, a bad one fails, and codex-proxy actually sent the requested `reasoning_effort`, before bulk collection.
- **Low yield** → not a deadline; log realized yield and scale up over time.
